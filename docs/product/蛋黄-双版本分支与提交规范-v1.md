## 核心结论

Tk/Python 和 Vue/Tauri 是两个平级产品版本。后续所有代码、文档、截图和发布记录都要标明目标线别，避免“在 Tk 分支改 Vue”或“在 Vue 分支覆盖 Tk”。

当前电脑处于 `tk/main`，默认先提交 Tk 线相关变更。另一台电脑的 Vue 进度应提交到 `vue/main` 或 `feature/vue/*`。`main` 是完整集成分支，最终要包含 Tk 和 Vue 两个版本的可公开源码、文档和共享资产索引。

## 分支规则

| 分支 | 用途 | 允许主要改动 |
| --- | --- | --- |
| `main` | 完整集成分支，包含 Tk 与 Vue 两个版本的可公开内容 | `src-prototype/`、`src-tauri-vue/`、`README.md`、`AGENTS.md`、`docs/product/`、公开资产索引 |
| `tk/main` | Tk/Python 可运行主线 | `src-prototype/`、Tk 文档、Tk QA 记录、Tk 打包说明 |
| `vue/main` | Vue/Tauri 产品化主线 | `src-tauri-vue/`、Vue/Tauri 文档、Vue/Tauri QA 记录 |
| `feature/tk/*` | Tk 功能、UI、修复批次 | Tk 边界内文件和对应 Tk 文档 |
| `feature/vue/*` | Vue/Tauri 功能、UI、修复批次 | Vue/Tauri 边界内文件和对应 Vue 文档 |
| `docs/shared/*` | 共享产品契约或仓库规则 | 不含运行数据和包产物的共享文档 |
| `backup/*` | 首次同步或风险操作前备份 | 只用于兜底，不长期开发 |

## 文档提交规则

任何版本优化都必须同步中文文档：

- Tk 改动：提交到 `tk/main` 或 `feature/tk/*`，文档标题或正文标明 `目标线别：Tk/Python 线`。
- Vue 改动：提交到 `vue/main` 或 `feature/vue/*`，文档标题或正文标明 `目标线别：Vue/Tauri 线`。
- 集成到 `main`：合并已验证的 Tk 与 Vue 公开内容，保留目录边界和文档线别，不把某一版本改成另一版本。
- 打包记录：源码仓库只提交发布说明、SHA256 和排除清单；实际 exe/zip 不提交，后续走 GitHub Releases。

文档命名优先使用：

```text
蛋黄-Tk功能名-文档类型-vN.md
蛋黄-Vue功能名-文档类型-vN.md
蛋黄-共享功能名-文档类型-vN.md
```

## 提交信息规则

Tk：

```text
feat(tk): ...
fix(tk): ...
docs(tk): ...
chore(tk): ...
```

Vue/Tauri：

```text
feat(vue): ...
fix(vue): ...
docs(vue): ...
chore(vue): ...
```

集成/共享：

```text
docs(shared): ...
chore(shared): ...
merge(tk): ...
merge(vue): ...
```

示例：

```powershell
git switch tk/main
git add src-prototype docs/product qa
git commit -m "docs(tk): add Tk front-end optimization plan"
git push origin tk/main
```

```powershell
git switch vue/main
git add src-tauri-vue docs/product
git commit -m "feat(vue): improve Tauri chat layout"
git push origin vue/main
```

## 提交前检查

每次提交前必须执行：

```powershell
git branch --show-current
git status --short
git diff --cached --stat
```

Tk 线检查：

```powershell
python -m py_compile data-dev/current-runtime/danhuang/run-danhuang-desktop-pet.py
python -m unittest discover -s src-prototype/modular/tests
python src-prototype/modular/validate_phase3.py --pet-dir data-dev/current-runtime/danhuang --json
```

Vue 线检查：

```powershell
cd src-tauri-vue
npm run type-check
npm run build
```

集成分支检查：

```powershell
git diff --cached --name-only
```

确认没有误加 `data-dev/`、`packages/`、`archives/`、`node_modules/`、`target/`、密钥、日志和本机隐私数据。

## GitHub 推送规则

- Tk 功能先合到 `tk/main`。
- Vue 功能先合到 `vue/main`。
- `main` 必须作为完整集成分支，定期合入 `tk/main` 和 `vue/main` 中已经验证、可公开提交的内容。
- 不使用 `git push --force` 覆盖另一台电脑的进度。
- 大包、exe、zip 不进代码仓库；需要共享时上传 GitHub Releases，并标明 `tk` 或 `vue`。
- Pull Request 标题必须带线别，例如 `[Tk]`、`[Vue]`、`[Shared]`。

推荐合入顺序：

```powershell
git switch main
git pull --ff-only
git merge --no-ff tk/main -m "merge(tk): integrate Tk line updates"
git merge --no-ff vue/main -m "merge(vue): integrate Vue line updates"
git push origin main
```

如果两边同时改了共享文档，先解决冲突并保留 Tk/Vue 两侧事实，再提交 `docs(shared): resolve version docs conflicts`。

## 冲突处理

- Tk 和 Vue 同时改共享文档时，先保留两边事实，再写 `差异与待补`，不要删掉另一版本信息。
- 如果另一台电脑 Vue 版本进度更快，先推到 `vue/main` 或 `backup/vue-*`，再从 `vue/main` 合入 `main`。
- 如果当前电脑 Tk 版本进度更快，先推到 `tk/main` 或 `backup/tk-*`，再从 `tk/main` 合入 `main`。
- `main` 解决的是“全部可公开内容在一个集成分支可见”，不是把 Tk 和 Vue 源码混到同一目录。

## 验收标准

- 当前分支和提交信息能看出版本线别。
- 文档正文能看出 `目标线别`。
- GitHub 上至少能区分 `tk/main` 和 `vue/main`。
- `main` 能看到 Tk 和 Vue 两个版本的公开源码与文档，但目录边界清晰。
- 仓库中不出现公开包禁止提交内容。
